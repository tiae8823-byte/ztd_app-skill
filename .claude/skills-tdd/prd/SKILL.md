# PRD 生成技能

通过自然语言对话，确定 MVP 边界，生成产品需求文档。

## 核心理念

**MVP 思维**：
- 不追求一次性想清楚所有细节
- 确定核心价值和最小可用形态
- 技术栈要确定（改造成本高）
- 功能可以边做边加
- **先调研再设计**：参考现有成熟方案，避免重复造轮子

**PRD 是活的**，不是合同。可以在 feat 过程中持续更新。

## 可用 MCP 工具

| MCP 工具 | 用途 |
|----------|------|
| `mcp__sequentialthinking__sequentialthinking` | 复杂需求分析 |
| `mcp__context7__resolve-library-id` + `mcp__context7__query-docs` | 查询技术框架文档 |
| `mcp__fetch__fetch` | 阅读参考网站 |
| WebSearch | 搜索类似项目、成熟方案、技术选型参考 |

## 执行流程

### 1. 理解需求

通过对话理解用户想要什么，不急着输出文档。

**关键问题**：
- 这个产品解决什么问题？
- 目标用户是谁？
- 核心价值是什么？（一句话说清楚）

---

### 2. 调研同类产品设计

在确定技术栈和架构之前，**先搜索同类产品**，了解主流的产品设计思路。

**调研重点**：产品层面（功能设计、交互模式、用户体验），不是技术实现。
- 搜索同类产品，了解它们的核心功能和交互设计
- 分析哪些设计值得借鉴，哪些可以优化
- 为 MVP 功能清单和界面设计提供参考

**调研方式**：
- WebSearch 搜索同类产品和竞品分析
- `mcp__fetch__fetch` 阅读产品介绍和评测文章

**输出**：
```markdown
调研了以下同类产品：
- [产品名称]：[核心功能]，[交互亮点]，[可借鉴的设计]
- [产品名称]：[核心功能]，[交互亮点]，[可借鉴的设计]

对本项目的启发：[总结]
```

> 快速了解主流产品怎么做的，重点借鉴功能设计和交互模式。

---

### 3. 确定技术栈（含 UI 方案）

根据需求和调研结果，**AI 分析后推荐**技术栈，不使用固定推荐表。

**UI 方案是技术栈的一部分**，在这一步一起确定，不单独拆分。

**分析维度**：
- **产品类型**：Web / 桌面 / 移动 / 跨平台 → 决定基础框架
- **用户规模和部署**：个人 / 团队 / 公开 → 决定后端和部署方式
- **团队技术背景**：熟悉什么语言和框架 → 减少学习成本
- **性能和离线需求**：是否需要离线、实时性要求 → 决定数据存储方式
- **交互复杂度**：简单 / 中等 / 复杂 → 决定前端方案

**UI 风格**：询问用户偏好（简洁/Material/企业），然后根据技术栈匹配对应的组件库或样式体系。

**推荐要求**：
- AI 推荐时**必须说明理由**（包括技术框架和 UI 方案）
- 给出 1-2 个方案供用户选择（如有多个合理方案）
- 用户确认后才确定
- 结合调研阶段的发现，优先采用经过验证的成熟方案

**输出示例**：
```markdown
推荐方案：
- 框架: WPF + .NET 8（你熟悉 C#，零学习成本）
- 数据库: SQLite（本地离线）
- UI 方案: 自定义品牌样式 + Material 风格
- 架构: MVVM + 依赖注入
```

---

### 4. 设计技术架构

**确定技术栈后，必须设计技术架构**，否则 feat 无法规范实现。

**设计原则**：
- 最适合 MVP：简单优先，避免过度设计
- 成本最低：使用成熟方案，减少学习成本
- 易于扩展：架构清晰，后期可迭代

**设计内容**：
- **项目目录结构**：清晰的分层划分
- **分层架构**：UI层 / 业务逻辑层 / 数据层
- **数据访问方式**：ORM / 原生SQL / 其他
- **状态管理**：如何管理应用状态
- **依赖注入**：如何管理依赖关系

**输出示例**：
```markdown
## 技术架构

### 项目目录结构
[根据实际技术栈生成]

### 分层架构
[根据实际技术栈生成架构图]

### 数据流转
[简述数据在各层之间的流转方式]
```

---

### 5. 确定 MVP 功能清单

讨论哪些是**必须有**的，哪些是**以后加**的。

**MVP 边界**：
- 最小可用形态是什么？
- 哪些功能可以后期迭代？

---

### 6. 生成 PRD.md

在 `docs/PRD.md` 创建文档：

```markdown
# PRD - [产品名称]

> 创建时间: [日期]
> 最后更新: [日期]

---

## 核心价值

[一句话描述产品解决什么问题]

---

## 参考方案

[调研阶段发现的参考项目和借鉴思路]

---

## 技术栈（含 UI 方案）

| 层级 | 技术 | 说明 |
|------|------|------|
| 框架 | [技术] | [选择理由] |
| 数据库 | [技术] | [选择理由] |
| UI 方案 | [组件库/样式体系] | [风格] |
| ... | ... | ... |

---

## 技术架构

### 项目目录结构
[目录结构]

### 分层架构
[架构图]

### 数据流转
[数据在各层之间的流转方式]

## 功能清单

### MVP 功能（必须实现）
```

### 7. 生成 feat 技能项目配置文件

PRD 确定技术栈后，**自动生成** feat 技能所需的 2 个项目专属配置文件。这些文件让 `/feat` 能直接使用正确的构建命令和样式系统。

**生成文件**：

| 文件 | 内容 | 数据来源 |
|------|------|---------|
| `.claude/skills/feat/project-config.md` | 构建/测试/运行命令、技术栈、样式系统、UI 检查清单、测试约定 | PRD 技术栈 + 技术架构 |
| `.claude/skills/feat/design-guide.md` | 颜色、字号、间距、组件样式清单 | PRD UI 方案 |

**project-config.md 模板**：
```markdown
# 项目技术配置

> 此文件由 /prd 自动生成。

## 构建与运行

| 操作 | 命令 |
|------|------|
| 构建 | `[根据技术栈生成]` |
| 运行 | `[根据技术栈生成]` |
| 单元测试 | `[根据技术栈生成]` |
| UI 测试 | `[根据技术栈生成]` |

## 技术栈
[从 PRD 技术栈表格复制]

## 样式系统
- 样式文件: [根据技术栈确定]
- 规则: 所有 UI 属性通过命名样式或 token 引用，禁止内联写死

## UI 一致性检查
[根据样式系统生成检查清单]

## 测试约定
[根据测试框架生成]
```

**design-guide.md 模板**：
```markdown
# 设计规范参考

## 颜色系统
[根据 UI 方案生成]

## Typography 层级
[根据技术栈生成合适的字号体系]

## 可用样式清单
[初始为空，随 /feat 迭代逐步补充]

## 提升共享样式流程
[通用流程，不变]
```

> **注意**：如果 `.claude/skills/feat/` 目录下已存在这些文件（如从模板复制过来的），不覆盖，而是提示用户是否更新。
> **testing-guide.md 不由 /prd 生成**：测试方法论是通用文件，随技能模块一起复制，不需要按项目定制。项目特定的测试约定（框架、命令）写在 project-config.md 的"测试约定"章节中。

---

### 8. 确认

生成后询问用户：

```
✅ PRD 已生成: docs/PRD.md
✅ feat 项目配置已生成:
   - .claude/skills/feat/project-config.md
   - .claude/skills/feat/design-guide.md

📋 内容摘要:
- 核心价值: [一句话]
- 参考方案: [调研的项目/方案]
- 技术栈: [框架 + UI 方案]
- 技术架构: [分层/模式]
- MVP 功能: [数量] 个

需要调整吗？没问题的话可以开始 /feat 实现功能。
```

## 对话示例

```
用户: /prd 个人生产力工具

AI: 个人生产力工具，具体想解决什么问题？是时间管理、任务管理、还是其他？

用户: 想法收集和任务管理，类似 GTD

AI: 了解，GTD 风格的想法收集 + 任务管理。目标平台是什么？Web、桌面、还是手机？

用户: 先做 Windows 桌面

AI: Windows 桌面应用。让我先调研一下同类产品的做法...

    [调研结果]
    调研了以下参考方案：
    - Todoist：跨平台，Web 优先，功能全面
    - TickTick：原生桌面端，支持番茄钟
    - Obsidian：Electron 架构，插件生态强

    桌面端常见的技术选型：
    - WPF / WinUI 3（C# 原生，性能好）
    - Electron / Tauri（跨平台，Web 技术栈）

    你熟悉哪些语言/框架？这样我好推荐合适的方案。

用户: 熟悉 C#

AI: 那推荐 WPF + .NET 8：
    - 你熟悉 C#，零学习成本
    - WPF 成熟稳定，桌面应用首选
    - SQLite 本地存储，离线可用
    - MVVM 架构，代码结构清晰

    这个方案可以吗？

用户: 可以

AI: 好，MVP 功能你想要哪些？

    建议包含：
    - 想法快速收集
    - 任务列表管理
    - 今日待办

    后续可加：
    - 番茄钟
    - 每周回顾
    - 统计分析

    你觉得呢？

用户: 可以，MVP 加个笔记功能

AI: [生成 PRD.md]

    ✅ PRD 已生成

    需要调整吗？
```

## 注意事项

- 不要一次性问太多问题，自然对话
- 技术栈推荐要说明理由，不用固定模板
- MVP 功能控制在 3-5 个核心功能
- 如果用户需求模糊，主动引导
- PRD 生成后允许随时调整
- **先调研再设计**：技术选型和架构设计前先看看别人怎么做的
