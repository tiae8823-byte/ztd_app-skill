# TDD 测试策略参考

> 测试驱动开发（TDD）方法论，适用于任何技术栈的 MVP 模块化开发。
> 项目特定的测试框架、命令和配置见 [project-config.md](project-config.md)。

## 核心理念

**测试先行，代码后写**：
- 测试定义"正确行为是什么"，代码实现"如何做到"
- 先写测试 = 先想清楚需求，再动手实现
- 测试是设计工具，不仅仅是验证工具
- 每个测试的维护成本不能超过它提供的信心价值

> 参考：Kent Beck "Test-Driven Development: By Example"、Martin Fowler "Refactoring"

## Red-Green-Refactor 详解

### Red（写失败的测试）

**目标**：定义一个具体的、可验证的行为期望。

**规则**：
- 测试必须运行并失败（编译错误也算失败）
- 每次只写**一个**测试（不要一次写完所有测试）
- 测试名称描述期望行为：`CreateIdea_WithEmptyContent_ShouldReject`
- 先写断言（Assert），再补充前置步骤（Arrange, Act）

**常见错误**：
- 写了测试但没运行确认失败 → 可能测试本身有 Bug
- 一次写太多测试 → 失去"小步"节奏
- 测试描述太模糊 → 不知道在验证什么

### Green（写最少的代码通过测试）

**目标**：用最简单的方式让测试从 Red 变 Green。

**规则**：
- 只写让当前失败测试通过的代码，**不多写**
- 允许"作弊"（硬编码返回值）—— 后续测试会逼你写真正的逻辑
- 不要提前优化，不要提前抽象
- 代码可以丑，但必须让测试通过

**常见错误**：
- 一口气写完所有实现 → 失去 TDD 的设计引导作用
- 过早优化 → 增加不必要的复杂度
- 跳过运行测试 → 不知道代码是否真的让测试通过

### Refactor（重构，保持绿色）

**目标**：改善代码结构，不改变行为。

**"两顶帽子"原则**（Martin Fowler）：

为什么 Green 和 Refactor 要分开？因为人的大脑（和 AI）不能同时追求两个目标：
1. 让代码**正确运行**（Green 阶段的帽子）
2. 让代码**结构优雅**（Refactor 阶段的帽子）

一次只戴一顶帽子。Green 阶段允许写丑代码，Refactor 阶段只改结构不改行为。这个分离让每一步都更简单、更安全。

**规则**：
- 每次小改动后运行测试，确保仍然绿色
- 消除重复代码（DRY）
- 改善命名和结构
- 不添加新功能
- 如果重构导致测试失败，立即撤销

**重构清单**：
- [ ] 有没有重复代码可以提取为方法？
- [ ] 方法名是否清晰表达意图？
- [ ] 方法是否过长（>20 行）需要拆分？
- [ ] 类的职责是否单一？
- [ ] 有没有魔法数字需要提取为常量？

## TDD 在 DTBRVC 各阶段的应用

| 阶段 | TDD 活动 | 交付物 |
|------|---------|--------|
| **Design** | 定义测试意图（5-8 条具体的行为期望） | 测试意图清单 |
| **Test** | 编写测试代码，确认全部 Red | 测试文件（全 Red） |
| **Build** | 逐个让测试从 Red 变 Green | 实现代码（全 Green） |
| **Refactor** | 重构代码，保持测试 Green | 整洁的代码 |
| **Verify** | 运行全部测试 + 手动冒烟 | 全绿测试 + 手动确认 |
| **Commit** | 确认无回归后提交 | git commit |

### Design 阶段的测试意图

测试意图比传统方式更详细，因为下一步就要变成测试代码：

```markdown
测试意图（每条对应 1-2 个测试用例）：
1. 输入有效内容 → 创建成功，数据持久化
2. 输入为空 → 拒绝创建，返回错误/不操作
3. 创建后 → 列表立即包含新条目
4. 超长内容 → 正常保存（边界条件）
5. 特殊字符 → 正常处理（防御性）
6. 删除 → 数据库同步清除
7. 重启后 → 数据仍然存在（持久化验证）
```

### Test 阶段的编写顺序

1. **从最简单的正向路径开始**（如：创建成功）
2. 再写负向路径（如：空内容拒绝）
3. 再写边界条件（如：超长内容）
4. 最后写集成场景（如：创建后列表更新）

每写一个测试：运行 → 确认 Red → 继续下一个。

### Build 阶段的实现顺序

1. **从让第一个测试通过开始**
2. 每通过一个测试，运行全部测试确认没有回归
3. 如果新代码导致之前的测试失败，优先修复
4. 所有测试 Green 后，进入 Refactor

## 测试分层（钻石型）

独立开发者的 MVP 项目推荐「钻石型」分布：

```
        ╱ ╲
       ╱ UI ╲          ← 少量，只覆盖关键路径（5-10%）
      ╱ 自动化 ╲
     ╱─────────╲
    ╱  集成测试  ╲      ← 核心，覆盖模块协作（15-20%）
   ╱─────────────╲
    ╲  单元测试  ╱      ← 适量，覆盖纯逻辑和边界（70-80%）
     ╲─────────╱
      ╲ 静态  ╱         ← 编译器 + 类型系统（零成本）
       ╲────╱
```

### 单元测试（Service/Manager 层）

**测什么**：
- 业务逻辑规则（验证、状态转换、数据转换）
- 边界条件（空值、极长内容、不存在的 ID）
- 防御性逻辑（死循环上限、异常处理）

**不测什么**：
- 简单的 CRUD 透传
- getter/setter
- 框架自动生成的代码

**TDD 中的角色**：Red-Green-Refactor 的主要战场。每个功能 6-10 个单元测试。

### 集成测试（ViewModel 层 —— 最有价值）

ViewModel 测试天然是集成测试：验证 ViewModel + Manager + Repository + 数据库 的完整链路。

**测什么**：
- 用户操作完整流程（输入 → 命令 → 状态变化 → 数据持久化）
- 跨模块数据流转
- 状态一致性（添加后列表更新、删除后列表清理）

**TDD 中的角色**：写在单元测试之后，验证各层组装是否正确。每个功能 5-8 个集成测试。

### UI 自动化测试（选择性使用）

**只测关键路径**：
- 启动不崩溃
- 核心功能的 happy path
- 跨页面数据流转

**TDD 中的角色**：在 Verify 阶段补充，不在 Red-Green 循环中。每个功能 2-4 个。

## AI 辅助 TDD 的特别规则

AI 同时写测试和代码时的额外风险：

| 反模式 | 描述 | TDD 防御 |
|--------|------|---------|
| 先写代码再补测试 | 失去 TDD 的设计驱动作用 | **严格 Red 先行**：必须先确认测试失败 |
| 测试为代码量身定做 | 测试只验证当前实现，不验证正确行为 | **人类审查测试意图**：Design 阶段确认 |
| 过度 Mock | Mock 一切导致测试不验证真实行为 | 优先用真实依赖（如内存数据库） |
| 跳过 Red 确认 | 没确认测试失败就开始写代码 | 强制 Red 确认步骤 |

**关键规则**：
1. **测试意图人类确认**：Design 阶段的测试意图必须用户 Y/N
2. **Red 确认不可跳过**：测试必须先运行并失败
3. **集成测试优先，少用 Mock**：保持完整依赖链的真实性
4. **人类审查断言**：检查每个 Assert 验证的是「正确行为」还是「当前行为」

### Bug 修复时的上下文污染

新功能的 TDD 流程天然没有此问题（Test 阶段时实现代码不存在）。但 **Bug 修复和功能修改** 时需要注意：

| 场景 | 风险 | 防御 |
|------|------|------|
| 新功能（TDD） | 无 | Test 先于 Build，实现不存在 |
| 修改已有功能 | AI 看到旧实现，测试可能迁就旧行为 | 基于"期望行为"而非"当前行为"写断言 |
| Bug 修复 | AI 看到 bug 代码，可能验证错误行为 | 先描述"正确应该怎样"，再写测试 |

**防御措施**：写 Bug 修复测试时，先用一句话描述"正确行为应该是什么"，再基于这个描述写断言。不要参考现有代码的行为。

### 测试也可以删除

Kent Beck："如果同一个行为被多种方式测试，那就是耦合。耦合有成本。"

**删除测试的场景**：
- 重构后，旧测试验证的行为已被更高层测试覆盖
- 多个测试验证同一件事（冗余）
- 测试维护成本 > 提供的信心价值
- 闪烁测试（不稳定，时过时不过）→ 修复或删除，不要留着

**不删除的底线**：删除测试后，核心业务逻辑的覆盖率不能降低。

## 增量测试策略

### 三层防御体系

```
功能 N 完成
    │
    ├─ 第一道防线：运行全部单元测试（秒级）
    │   → 功能 N 破坏了已有 Manager 逻辑？这里抓住
    │
    ├─ 第二道防线：运行全部 ViewModel 测试（秒级）
    │   → 功能 N 修改共享服务导致其他 ViewModel 行为变化？这里抓住
    │
    └─ 第三道防线：运行 UI 测试 + 手动冒烟测试（分钟级）
        → 端到端验证
```

### 增量集成测试检查清单

每完成一个新功能后：

```
□ 1. 编译通过
□ 2. 全部测试通过（单元 + 集成，不过滤）
□ 3. 新功能的 UI 测试通过
□ 4. 手动冒烟测试：
    □ 4a. 导航到所有页面，确认不崩溃
    □ 4b. 在每个已有页面执行一次核心操作
    □ 4c. 验证新功能与已有功能的数据流转（如有交叉）
□ 5. 关闭应用、重新打开，确认数据持久化正常
```

## TDD vs 传统 DBTVC 对比

| 维度 | 传统 DBTVC | TDD (DTBRVC) |
|------|-----------|--------------|
| 测试时机 | 代码写完后 | 代码写之前 |
| 测试角色 | 验证工具 | 设计工具 + 验证工具 |
| 代码质量 | 依赖开发者自觉 | 测试驱动，天然可测试 |
| 过度实现 | 容易写多余代码 | 只写测试要求的代码 |
| 重构信心 | 可能破坏功能 | 测试保护，放心重构 |
| 学习曲线 | 低 | 中等（需要练习） |
| 适用场景 | 快速原型、UI 探索 | 业务逻辑复杂的功能 |

## 何时投入更多测试、何时跳过

### 值得重点测试（TDD 全力投入）

- 涉及数据持久化的逻辑
- 跨模块的数据流转
- 有条件分支的业务规则
- 用户可能提供异常输入的入口

### 可以简化测试（降低 TDD 粒度）

- 纯 UI 展示逻辑（颜色、间距）—— 手动看一眼即可
- 框架自动生成的代码
- 简单的一行透传方法
- 第三方库的行为

## 常见陷阱

| 陷阱 | 后果 | 解决 |
|------|------|------|
| 跳过 Red 直接写代码 | 失去 TDD 核心价值 | 严格执行 Red 确认 |
| 测试粒度太大 | Red-Green 循环太慢 | 拆分为更小的测试 |
| 测试粒度太小 | 测试太多，维护成本高 | 按行为而非方法来组织测试 |
| Refactor 时加功能 | 混淆"改结构"和"加功能" | Refactor 只改结构不改行为 |
| 测试间共享状态 | 运行顺序影响结果 | 每个测试独立数据 |
| 追求 100% 覆盖 | 收益递减，浪费时间 | 业务逻辑 70-80% 即可 |

## 手动测试清单

| 测试项 | 测试步骤 | 预期结果 |
|--------|---------|---------
| **1. 启动测试** | 运行应用 | 窗口正常显示 |
| **2. 基础功能** | 执行功能主要操作 | 按预期工作 |
| **3. UI 样式** | 检查按钮、输入框等 | 符合设计规范 |
| **4. 交互反馈** | 悬停、点击、聚焦 | 有正确的视觉反馈 |
| **5. 异常输入** | 空值、超长、特殊字符 | 正确处理 |
| **6. 数据持久化** | 关闭重启 | 数据保留 |
