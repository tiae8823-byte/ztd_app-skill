## 正确的渐进式披露结构

### 理想 SKILL.md（< 80 行）

```markdown
---
name: feat
description: Implement single feature using Design-Intent-Build-Verify-Commit cycle.
---

# 功能实现技能

采用 **Design-Intent-Build-Verify-Commit** 循环实现**单个**功能。

## 核心理念

- **一次只实现一个功能**
- **测试意图先行** - 人工定义预期，AI 实现代码
- **调研贯穿全流程** - GitHub > 博客 > 官方文档
- **分层测试** - CRITICAL 冒烟测试，NON-CRITICAL 手动测试

## 流程

```
Design → Intent → Build → Verify → Commit
  ↓        ↓        ↓        ↓
调研UI   定义预期  实现     验证
         (人工)          (分层)
```

| 阶段 | 核心任务 | 强制检查点 |
|------|---------|-----------|
| **Design** | UI布局 + 复杂度评估 | **必须**输出 Given-When-Then + 用户确认 |
| **Intent** | 测试意图确认（仅 CRITICAL） | **必须**人工确认 |
| **Build** | 调研方案 → 编码实现 | **必须**告知参考来源 |
| **Verify** | 编译 + 分层测试 | **必须**运行检查清单 |
| **Commit** | 提交代码 | **必须**更新PRD + 清理进度 |

## 详细指南

- **完整工作流程**: [references/workflow-guide.md](references/workflow-guide.md)
- **测试意图先行**: [references/test-intent-first.md](references/test-intent-first.md)
- **分层测试策略**: [references/testing-guide.md](references/testing-guide.md)
- **设计规范**: [design-guide.md](design-guide.md)
- **项目配置**: [project-config.md](project-config.md)

## 关键原则

1. **必须使用共享样式** - 禁止内联 ⚠️
2. **测试意图先行** - 人工定义，AI 实现
3. **调研先行** - GitHub > 博客 > 官方文档
4. **分层测试** - CRITICAL 冒烟测试，NON-CRITICAL 手动测试
5. **及时保存进度** - 每个阶段后更新 feat-progress.json

## 进度管理

进度保存在 `.claude/feat-progress.json`

| Status | 说明 |
|--------|------|
| `design` | 正在设计 |
| `intent` | 定义测试意图（CRITICAL） |
| `build` | 正在实现 |
| `verify` | 正在验证 |
| `commit` | 准备提交 |
```

**行数**：~70 行

**特点**：
- ✅ 极简概览
- ✅ 所有详细内容都在 references/
- ✅ 符合"目录而非百科全书"
