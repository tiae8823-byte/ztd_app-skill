# PRD 生成技能

通过自然语言对话，确定 MVP 边界，生成产品需求文档。

## 核心理念

**MVP 思维**：
- 不追求一次性想清楚所有细节
- 确定核心价值和最小可用形态
- 技术栈要确定（改造成本高）
- 功能可以边做边加

**PRD 是活的**，不是合同。可以在 feat 过程中持续更新。

## 可用 MCP 工具

| MCP 工具 | 用途 |
|----------|------|
| `mcp__sequential-thinking__sequentialthinking` | 复杂需求分析 |
| `mcp__github__search_repositories` | 搜索类似项目参考 |
| `mcp__context7__resolve-library-id` + `mcp__context7__query-docs` | 查询技术框架文档 |
| `mcp__web_reader__webReader` | 阅读参考网站 |

## 执行流程

### 1. 理解需求

通过对话理解用户想要什么，不急着输出文档。

**关键问题**：
- 这个产品解决什么问题？
- 目标用户是谁？
- 核心价值是什么？（一句话说清楚）

### 2. 确定技术栈

根据业务特点推荐技术栈，不使用预设模板。

**讨论要点**：
- 业务复杂度 → 决定后端复杂度
- 用户规模 → 决定部署方式
- 团队情况 → 决定技术选型
- 交互复杂度 → 决定前端框架

**常见推荐**（仅供参考，根据实际情况调整）：

| 场景 | 推荐方案 |
|------|----------|
| 简单内容站 | Next.js SSG + Markdown |
| 中型应用 | Next.js + FastAPI + PostgreSQL |
| 快速原型 | Next.js + Supabase |
| 工具类 | Electron / Tauri |

### 3. 设计技术架构

**确定技术栈后，必须设计技术架构**，否则 feat 无法规范实现。

**设计原则**：
- 最适合 MVP：简单优先，避免过度设计
- 成本最低：使用成熟方案，减少学习成本
- 易于扩展：架构清晰，后期可迭代

**设计内容**：
- **项目目录结构**：清晰的分层划分
- **分层架构**：UI层 / 业务逻辑层 / 数据层
- **数据访问方式**：ORM / 原生SQL / 其他
- **状态管理**：如何管理应用状态
- **依赖注入**：如何管理依赖关系

**输出示例**：
```markdown
## 技术架构

### 项目目录结构
```
/src
  /Models        # 数据模型
  /Data          # 数据访问层
  /Services      # 业务逻辑层
  /ViewModels    # 视图模型
  /Views         # 界面视图
```

### 分层架构
```
┌─────────────────────────────────┐
│         UI 层 (Views)            │
└─────────────┬───────────────────┘
              │
┌─────────────▼───────────────────┐
│      ViewModel 层 (MVVM)         │
└─────────────┬───────────────────┘
              │
┌─────────────▼───────────────────┐
│      Service 层 (业务逻辑)       │
└─────────────┬───────────────────┘
              │
┌─────────────▼───────────────────┐
│      Data 层 (数据访问)          │
└─────────────┬───────────────────┘
              │
┌─────────────▼───────────────────┐
│      SQLite 数据库               │
└─────────────────────────────────┘
```

### 数据流转
[简述数据在各层之间的流转方式]
```

**技术栈选择指南**：

| 技术栈 | 推荐架构 | 理由 |
|--------|----------|------|
| C# + WinUI 3 | MVVM + 依赖注入 | WinUI 3 官方推荐 |
| React Native + Expo | Redux/Context + 组件化 | React 生态最佳实践 |
| Next.js | Server Components + API Routes | Next.js 最新特性 |

### 4. 确定 UI 风格

简单询问偏好，直接用现成组件库：

- 苹果风格 → shadcn/ui（默认）
- 谷歌风格 → Material UI
- 企业风格 → Ant Design

### 5. 设计技术架构

根据确定的技术栈，设计清晰的技术架构，确保 feat 实现有据可依。

### 6. 确定 MVP 功能清单

讨论哪些是**必须有**的，哪些是**以后加**的。

**MVP 边界**：
- 最小可用形态是什么？
- 哪些功能可以后期迭代？

### 7. 生成 PRD.md

在 `docs/PRD.md` 创建文档：

```markdown
# PRD - [产品名称]

> 创建时间: [日期]
> 最后更新: [日期]

---

## 核心价值

[一句话描述产品解决什么问题]

---

## 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | [技术] | [理由] |
| 后端 | [技术] | [理由] |
| 数据库 | [技术] | [理由] |
| UI组件库 | [组件库] | [风格] |
| 部署 | [方式] | [理由] |

---

## 技术架构

### 项目目录结构
```
[目录结构]
```

### 分层架构
```
[架构图]
```

### 数据流转
[数据在各层之间的流转方式]

---

## 功能清单

### MVP 功能（必须实现）

- [ ] [功能1]
- [ ] [功能2]
- [ ] [功能3]

### 后续功能（未来迭代）

- [ ] [功能A]
- [ ] [功能B]

---

## 备注

[其他需要记录的信息]
```

### 7. 确认

生成后询问用户：

```
✅ PRD 已生成: docs/PRD.md

📋 内容摘要:
- 核心价值: [一句话]
- 技术栈: [前端] + [后端]
- 技术架构: [分层/模式]
- MVP 功能: [数量] 个
- UI 风格: [组件库]

需要调整吗？没问题的话可以开始 /feat 实现功能。
```

### 8. 设计技术架构（可选补充）

## 对话示例

```
用户: /prd 博客系统

AI: 好的，博客系统。个人博客还是多用户平台？

用户: 个人博客

AI: 个人博客。主要用途是什么？记录技术文章、生活随笔、还是其他？

用户: 技术文章为主

AI: 技术博客。写作体验上有什么要求？Markdown 支持重要吗？

用户: 要支持 Markdown，实时预览

AI: 了解。流量预期怎样？

用户: 主要是自己看，偶尔分享出去

AI: 这样的话，建议：
    - Next.js（静态生成，速度快，免费部署）
    - MDX 管理文章（不需要数据库）
    - shadcn/ui（简洁风格）
    - 部署 Vercel

    后续想加评论的话可以用 Giscus。这个方案可以吗？

用户: 可以

AI: 好，整理 PRD。MVP 功能你想要哪些？

    建议包含：
    - 文章列表
    - 文章详情
    - 代码高亮

    后续可加：
    - 搜索
    - 标签分类
    - 评论

    你觉得呢？

用户: MVP 加个关于页面，其他的可以

AI: [生成 PRD.md]

    ✅ PRD 已生成

    需要调整吗？
```

## 注意事项

- 不要一次性问太多问题，自然对话
- 技术栈推荐要说明理由
- MVP 功能控制在 3-5 个核心功能
- 如果用户需求模糊，主动引导
- PRD 生成后允许随时调整
