# PRD 生成详细工作流程

> 完整的 PRD 生成流程详细说明

---

## 流程概览

```
用户: /prd [产品描述]
  │
  ▼
理解需求 ←→ 自然对话
  │
  ▼
调研同类产品
  │
  ▼
确定技术栈（含UI方案）
  │
  ▼
设计技术架构
  │
  ▼
确定MVP功能清单
  │
  ▼
生成 PRD.md
  │
  ▼
生成 feat 项目配置
  │
  ▼
确认完成
```

---

## 各步骤详解

### 1. 理解需求

通过对话理解用户想要什么，不急着输出文档。

**关键问题**：
- 这个产品解决什么问题？
- 目标用户是谁？
- 核心价值是什么？（一句话说清楚）

**对话原则**：
- 不要一次性问太多问题
- 自然对话，根据用户回答追问
- 如果需求模糊，主动引导

---

### 2. 调研同类产品设计

在确定技术栈和架构之前，**先搜索同类产品**，了解主流的产品设计思路。

**调研重点**：产品层面（功能设计、交互模式、用户体验），不是技术实现。
- 搜索同类产品，了解它们的核心功能和交互设计
- 分析哪些设计值得借鉴，哪些可以优化
- 为 MVP 功能清单和界面设计提供参考

**调研方式**：
- WebSearch 搜索同类产品和竞品分析
- `mcp__fetch__fetch` 阅读产品介绍和评测文章

**输出**：
```markdown
调研了以下同类产品：
- [产品名称]：[核心功能]，[交互亮点]，[可借鉴的设计]
- [产品名称]：[核心功能]，[交互亮点]，[可借鉴的设计]

对本项目的启发：[总结]
```

> 快速了解主流产品怎么做的，重点借鉴功能设计和交互模式。

---

### 3. 确定技术栈（含 UI 方案）

根据需求和调研结果，**AI 分析后推荐**技术栈，不使用固定推荐表。

**UI 方案是技术栈的一部分**，在这一步一起确定，不单独拆分。

**分析维度**：
- **产品类型**：Web / 桌面 / 移动 / 跨平台 → 决定基础框架
- **用户规模和部署**：个人 / 团队 / 公开 → 决定后端和部署方式
- **团队技术背景**：熟悉什么语言和框架 → 减少学习成本
- **性能和离线需求**：是否需要离线、实时性要求 → 决定数据存储方式
- **交互复杂度**：简单 / 中等 / 复杂 → 决定前端方案

**UI 风格**：询问用户偏好（简洁/Material/企业），然后根据技术栈匹配对应的组件库或样式体系。

**推荐要求**：
- AI 推荐时**必须说明理由**（包括技术框架和 UI 方案）
- 给出 1-2 个方案供用户选择（如有多个合理方案）
- 用户确认后才确定
- 结合调研阶段的发现，优先采用经过验证的成熟方案

**输出示例**：
```markdown
推荐方案：
- 框架: WPF + .NET 8（你熟悉 C#，零学习成本）
- 数据库: SQLite（本地离线）
- UI 方案: 自定义品牌样式 + Material 风格
- 架构: MVVM + 依赖注入
```

---

### 4. 设计技术架构

**确定技术栈后，必须设计技术架构**，否则 feat 无法规范实现。

**设计原则**：
- 最适合 MVP：简单优先，避免过度设计
- 成本最低：使用成熟方案，减少学习成本
- 易于扩展：架构清晰，后期可迭代

**设计内容**：
- **项目目录结构**：清晰的分层划分
- **分层架构**：UI层 / 业务逻辑层 / 数据层
- **数据访问方式**：ORM / 原生SQL / 其他
- **状态管理**：如何管理应用状态
- **依赖注入**：如何管理依赖关系

**输出示例**：
```markdown
## 技术架构

### 项目目录结构
[根据实际技术栈生成]

### 分层架构
[根据实际技术栈生成架构图]

### 数据流转
[简述数据在各层之间的流转方式]
```

---

### 5. 确定 MVP 功能清单

讨论哪些是**必须有**的，哪些是**以后加**的。

**MVP 边界**：
- 最小可用形态是什么？
- 哪些功能可以后期迭代？

**建议**：MVP 控制在 3-5 个核心功能

---

### 6. 生成 PRD.md

在 `docs/PRD.md` 创建文档：

```markdown
# PRD - [产品名称]

> 创建时间: [日期]
> 最后更新: [日期]

---

## 核心价值

[一句话描述产品解决什么问题]

---

## 参考方案

[调研阶段发现的参考项目和借鉴思路]

---

## 技术栈（含 UI 方案）

| 层级 | 技术 | 说明 |
|------|------|------|
| 框架 | [技术] | [选择理由] |
| 数据库 | [技术] | [选择理由] |
| UI 方案 | [组件库/样式体系] | [风格] |
| ... | ... | ... |

---

## 技术架构

### 项目目录结构
[目录结构]

### 分层架构
[架构图]

### 数据流转
[数据在各层之间的流转方式]

---

## 功能清单

### MVP 功能（必须实现）

- [ ] [功能1]
- [ ] [功能2]
- [ ] [功能3]

### 后续功能（未来迭代）

- [ ] [功能A]
- [ ] [功能B]

---

## 备注

[其他需要记录的信息]
```

---

### 7. 生成 feat 技能项目配置文件

PRD 确定技术栈后，**自动生成** feat 技能所需的 2 个项目专属配置文件。这些文件让 `/feat` 能直接使用正确的构建命令和样式系统。

**生成文件**：

| 文件 | 内容 | 数据来源 |
|------|------|---------|
| `.claude/skills/feat/project-config.md` | 构建/测试/运行命令、技术栈、样式系统、UI 检查清单、测试约定 | PRD 技术栈 + 技术架构 |
| `.claude/skills/feat/design-guide.md` | 颜色、字号、间距、组件样式清单 | PRD UI 方案 |

> **注意**：如果 `.claude/skills/feat/` 目录下已存在这些文件（如从模板复制过来的），不覆盖，而是提示用户是否更新。

---

### 8. 确认

生成后询问用户：

```
✅ PRD 已生成: docs/PRD.md
✅ feat 项目配置已生成:
   - .claude/skills/feat/project-config.md
   - .claude/skills/feat/design-guide.md

📋 内容摘要:
- 核心价值: [一句话]
- 参考方案: [调研的项目/方案]
- 技术栈: [框架 + UI 方案]
- 技术架构: [分层/模式]
- MVP 功能: [数量] 个

需要调整吗？没问题的话可以开始 /feat 实现功能。
```
