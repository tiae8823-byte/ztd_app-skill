# TDD 意图模式

测试驱动开发的核心：先定义预期行为，再写代码实现。

## 为什么需要

**问题**：AI 看着代码写测试 → 测试验证的是"当前行为"，不一定是"正确行为"

**解决**：测试意图先行 → 人工确认预期 → AI 写测试 → AI 写代码

## 流程

```
Design → 列出测试意图（3-5条场景）
   ↓
Intent → 人工审查、补充、确认
   ↓
Test → AI 根据意图编写测试框架
   ↓
Build → 参考测试意图写代码
   ↓
Verify → 运行测试验证
```

## 测试意图模板

```markdown
测试意图：
1. [场景]: [输入/条件] → [预期结果]
2. [场景]: [输入/条件] → [预期结果]

边界条件：
- [边界]: [描述]

异常处理：
- [异常情况]: [预期行为]
```

## 示例：番茄钟计时

```markdown
测试意图：
1. 开始计时: 点击开始 → 倒计时启动，每秒更新
2. 暂停功能: 点击暂停 → 倒计时停止
3. 计时完成: 时间到 0 → 触发完成事件
4. 10秒内可撤回: 剩余<10秒 → 显示撤回按钮

边界条件：
- 5分钟内只能撤回一次
- 已完成计时无法撤回

异常处理：
- 暂停超过30分钟 → 自动结束
- 应用重启 → 恢复计时状态
```

## 何时使用

**必须使用：**
- 时间/日期逻辑
- 状态流转
- 复杂计算
- 跨模块同步
- 条件分支业务规则

**不需要：**
- 纯 UI 调整
- 简单列表展示
- 静态内容
- 简单导航

## 关键原则

1. 测试意图用中文写（作为需求文档）
2. 每个意图一个场景
3. 明确输入和预期
4. 覆盖边界条件
5. **人工必须确认**（防止"看着代码写测试"）
