# 测试策略参考

> 通用测试方法论，适用于任何技术栈的 MVP 模块化开发。
> 项目特定的测试框架、命令和配置见 [project-config.md](project-config.md)。

## 核心理念

**投资回报率驱动测试**：
- 测试是安全网，不是交付门槛
- 写测试，别写太多，主要写集成测试
- 测试越接近用户的真实使用方式，信心越大
- 每个测试的维护成本不能超过它提供的信心价值

> 参考：Kent C. Dodds "Testing Trophy"、Google SMURF 框架

---

## 测试分层（钻石型）

独立开发者的 MVP 项目推荐「钻石型」分布，而非传统金字塔：

```
        ╱ ╲
       ╱ UI ╲          ← 少量，只覆盖关键路径（5-10%）
      ╱ 自动化 ╲
     ╱─────────╲
    ╱  集成测试  ╲      ← 核心，覆盖模块协作（15-20%）
   ╱─────────────╲
    ╲  单元测试  ╱      ← 适量，覆盖纯逻辑和边界（70-80%）
     ╲─────────╱
      ╲ 静态  ╱         ← 编译器 + 类型系统（零成本）
       ╲────╱
```

**核心原则**：能在低层验证的，不要推到高层测试。

## 各层测试策略

### 单元测试（Service/Manager 层）

**测什么**：
- 业务逻辑规则（验证、状态转换、数据转换）
- 边界条件（空值、极长内容、不存在的 ID）
- 防御性逻辑（死循环上限、异常处理）

**不测什么**：
- 简单的 CRUD 透传
- getter/setter
- 框架自动生成的代码

**每个功能参考**：6-10 个单元测试

### 集成测试（ViewModel 层 —— 最有价值）

ViewModel 测试天然是集成测试：验证 ViewModel + Manager + Repository + 数据库 的完整链路。

**测什么**：
- 用户操作完整流程（输入 → 命令 → 状态变化 → 数据持久化）
- 跨模块数据流转
- 状态一致性（添加后列表更新、删除后列表清理）

**每个功能参考**：7-11 个集成测试

### UI 自动化测试（选择性使用）

**只测关键路径**：
- 启动不崩溃
- 核心功能的 happy path
- 跨页面数据流转

**不测**：
- 精确的 UI 布局和样式（用手动测试代替）
- 每个边界条件的 UI 表现
- 微交互（悬停、聚焦）

**每个功能参考**：2-4 个 UI 测试

## DBTVC 各阶段的测试职责

| 阶段 | 测试相关活动 | 交付物 |
|------|-------------|--------|
| **Design** | 识别需要测试的核心路径和跨模块交叉点 | 测试意图清单（3-5 条场景描述） |
| **Build** | 无测试活动，专注实现（保持代码可测试性） | 功能代码 |
| **Test** | 编写单元测试 + ViewModel 集成测试 | 测试代码文件 |
| **Verify** | 运行全部测试 + UI 测试 + 手动冒烟 | 全绿测试结果 + 手动确认 |
| **Commit** | 确认无回归后提交 | git commit |

### Design 阶段的测试意图

在设计 UI 布局时，同时识别 3-5 条需要测试的核心场景：

```markdown
测试意图：
1. 当用户输入空内容时，不应该创建新条目
2. 当用户将想法转为待办时，想法消失，待办列表出现这条数据
3. 快速完成应该创建已完成状态的任务
```

### Test 阶段的编写顺序

1. 先写单元测试（Service/Manager 的业务逻辑）
2. 再写 ViewModel 集成测试（用户操作流程）
3. 最后写/更新 UI 测试（仅关键路径）
4. 运行全部已有测试确认无回归

## 增量测试策略

### 三层防御体系

```
功能 N 完成
    │
    ├─ 第一道防线：运行全部单元测试（秒级）
    │   → 功能 N 破坏了已有 Manager 逻辑？这里抓住
    │
    ├─ 第二道防线：运行全部 ViewModel 测试（秒级）
    │   → 功能 N 修改共享服务导致其他 ViewModel 行为变化？这里抓住
    │
    └─ 第三道防线：运行 UI 测试 + 手动冒烟测试（分钟级）
        → 端到端验证
```

### 增量集成测试检查清单

每完成一个新功能后：

```
□ 1. 编译通过
□ 2. 全部测试通过（单元 + 集成，不过滤）
□ 3. 新功能的 UI 测试通过
□ 4. 手动冒烟测试：
    □ 4a. 导航到所有页面，确认不崩溃
    □ 4b. 在每个已有页面执行一次核心操作
    □ 4c. 验证新功能与已有功能的数据流转（如有交叉）
□ 5. 关闭应用、重新打开，确认数据持久化正常
```

### 跨模块回归测试

当新功能与已有功能有数据交叉时，在已有测试文件中补充回归测试点。

## AI 辅助开发的测试规则

AI 同时写代码和测试时的常见问题：

| 反模式 | 描述 | 防御 |
|--------|------|------|
| 验证即确认 | 测试只确认 AI 写的代码行为，不管行为是否正确 | 人类审查每个 Assert |
| 过度 Mock | Mock 一切导致测试不验证真实行为 | 优先用真实依赖（如内存数据库） |
| 作弊通过 | 注释断言或降低验证标准让测试通过 | 检查测试是否真的验证了预期 |

**关键规则**：
1. **人类审查断言**：AI 生成测试后，检查每个 `Assert` 验证的是「正确行为」还是仅仅「当前行为」
2. **集成测试优先，少用 Mock**：保持完整依赖链的真实性
3. **人写测试意图，AI 写实现**：开发者用中文描述测试场景，AI 来实现代码

## 手动测试清单

| 测试项 | 测试步骤 | 预期结果 |
|--------|---------|---------|
| **1. 启动测试** | 运行应用 | 窗口正常显示 |
| **2. 基础功能** | 执行功能主要操作 | 按预期工作 |
| **3. UI 样式** | 检查按钮、输入框等 | 符合设计规范 |
| **4. 交互反馈** | 悬停、点击、聚焦 | 有正确的视觉反馈 |
| **5. 异常输入** | 空值、超长、特殊字符 | 正确处理 |
| **6. 数据持久化** | 关闭重启 | 数据保留 |

## 何时投入更多测试、何时跳过

### 值得重点测试

- 涉及数据持久化的逻辑
- 跨模块的数据流转
- 有条件分支的业务规则
- 用户可能提供异常输入的入口

### 可以跳过测试

- 纯 UI 展示逻辑（颜色、间距）—— 手动看一眼即可
- 框架自动生成的代码
- 简单的一行透传方法
- 第三方库的行为

## 常见陷阱

| 陷阱 | 后果 | 解决 |
|------|------|------|
| 跳过测试「就这一次」 | 变成习惯，3 个功能后无覆盖 | 每个功能至少几个测试 |
| 测试错误的东西 | 虚假安全感，核心逻辑未覆盖 | 按风险排优先级 |
| 跨层重复测试 | 改一个规则三个测试都挂 | 每个行为只在最低层测试 |
| 闪烁测试 | 忽视测试失败 | 删除不稳定的测试 |
| 追求 100% 覆盖 | 收益递减，浪费时间 | 业务逻辑 70-80% 即可 |
| 测试间共享状态 | 运行顺序影响结果 | 每个测试独立数据 |
