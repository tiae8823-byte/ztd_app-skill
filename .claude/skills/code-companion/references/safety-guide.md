# 安全评估详则

AI 评估改动安全性时参考此文件。

## 通用安全分类

| 文件类型 | 等级 | 说明 |
|----------|------|------|
| 样式/主题文件 | 🟢 | 颜色和样式定义，改了立即见效 |
| 视图/模板 — 视觉部分 | 🟢 | 文字内容、间距、大小、背景色 |
| 视图/模板 — 绑定/事件 | 🟡 | 数据绑定、事件处理、路由 |
| 业务逻辑层 | 🟡 | 界面交互逻辑，改名可能断绑定 |
| 数据模型 | 🟡 | 数据结构，改字段可能影响存储 |
| 服务/核心逻辑 | 🔴 | 核心业务逻辑 |
| 数据访问层 | 🔴 | 数据库操作，改错会丢数据 |
| 启动/配置文件 | 🔴 | 应用入口和依赖配置，改错启动不了 |
| 项目构建配置 | 🔴 | 构建配置，改错编译不了 |

> 具体文件路径参考项目的 CLAUDE.md 和目录结构。

## 快速判断规则

改动前过一遍这 3 条：

1. **只改视觉？** 文字/颜色/大小/间距 → 🟢
2. **涉及数据绑定或事件处理？** → 🟡 需要确认对应逻辑存在
3. **涉及业务逻辑代码？** → 🟡🔴 建议用 /feat

## 🟢 安全改动清单

```
文字内容：按钮文字、标签文字、页面标题
颜色样式：背景色、前景色、引用主题中的颜色变量
尺寸间距：宽度、高度、边距、内边距、字号
布局比例：列宽、行高等布局分配
```

## 🟡 需确认的改动

```
数据绑定路径：改名要同步改对应逻辑层
事件/命令绑定：确保有对应处理方法
新增 UI 组件：可能需要逻辑层配合
路由/导航：确保目标页面存在
```

## 🔴 不执行，引导用 /feat

```
数据访问层的任何文件
服务层的业务逻辑
应用启动和依赖注入配置
数据库 schema 变更
新增页面/功能模块
```

## 出错时的回滚

告诉用户：
- `git checkout -- 文件路径` — 恢复单个文件
- `git stash` — 暂存所有改动
- `git stash pop` — 恢复暂存的改动
- 强调：**不用怕改坏，git 记着所有历史**
