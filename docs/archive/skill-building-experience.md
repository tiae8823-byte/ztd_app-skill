# 搭建 AI 辅助开发技能系统的经验

> 从零搭建 /prd + /feat 技能系统的完整经验记录。
> 适用于想要用 Claude Code 构建自己的开发流程的人。

---

## 一、核心设计理念

### 1.1 为什么需要技能系统？

AI 编码助手（Claude Code）很强大，但缺乏**流程约束**：
- 不知道先做什么后做什么
- 不知道项目的样式规范
- 不知道测试要求
- 每次对话都要重新说明

技能系统的本质是**把你的开发流程编码化**，让 AI 每次都按同一套流程执行。

### 1.2 设计原则

| 原则 | 说明 | 反面教材 |
|------|------|---------|
| **通用与专属分离** | 流程通用，配置项目专属 | 把项目配置写死在流程文件里 |
| **一次一个功能** | 小步快跑，完成即提交 | 一次实现三个功能 |
| **进度可中断恢复** | 每个阶段保存进度 | 必须一口气做完 |
| **UI 一致性强制** | 共享样式优先，禁止内联 | 每个页面自己定义颜色 |
| **文档是活的** | PRD 随时更新，不是合同 | 需求文档写完就不改了 |

## 二、技能架构

### 2.1 两个核心技能

```
/prd  →  确定"做什么"（需求、技术栈、架构）
/feat →  确定"怎么做"（一次一个功能的开发循环）
```

这两个技能的分工很清晰：
- `/prd` 只做决策，不写代码
- `/feat` 只写代码，不改架构

### 2.2 文件架构

```
.claude/skills/
├── workflow-overview.md     ← 总览（不改）
├── prd/
│   └── SKILL.md             ← PRD 技能定义（不改）
└── feat/
    ├── skill.md             ← 开发流程定义（不改）
    ├── testing-guide.md     ← 测试方法论（不改）
    ├── troubleshooting-guide.md  ← 问题排查（不改）
    ├── examples.md          ← 对话示例（不改）
    ├── project-config.md    ← 项目配置（/prd 生成）
    └── design-guide.md      ← 设计规范（/prd 生成）
```

**关键决策**：通用文件 vs 项目专属文件。

换项目时，只需要重新运行 `/prd` 生成 `project-config.md` 和 `design-guide.md`，其余文件直接复制。

### 2.3 为什么这样分？

这个决策经历了多次迭代：

**最初版本**：所有配置写在 `skill.md` 里
- 问题：换项目要改流程文件，容易改错

**第二版**：拆出 `project-config.md`
- 问题：测试策略也写在里面，但测试方法论是通用的

**最终版**：三层分离
- 流程（skill.md）= 怎么做 → 不改
- 方法论（testing-guide.md）= 测试理念 → 不改
- 配置（project-config.md）= 用什么工具 → /prd 生成

**经验教训**：
> 如果一个信息在不同项目中都一样，它就是"通用"的，应该独立成文件。
> 如果一个信息换项目就变，它就是"项目专属"的，应该由 /prd 生成。

## 三、开发循环的演进

### 3.1 传统版本（DBTVC）

```
Design → Build → Test → Verify → Commit
```

**优势**：简单直观，适合快速原型
**劣势**：测试后写，容易"迁就"代码

### 3.2 TDD 版本（DTBRVC）

```
Design → Test → Build → Refactor → Verify → Commit
```

**优势**：测试驱动设计，代码质量更高
**劣势**：学习曲线，纯 UI 功能不太适合

### 3.3 何时用哪个？

我的结论是：**两个版本都保留**，根据项目和功能类型选择。

| 场景 | 推荐版本 |
|------|---------|
| MVP 快速验证 | DBTVC（传统） |
| 业务逻辑复杂 | DTBRVC（TDD） |
| 纯 UI 功能 | DBTVC |
| 核心数据流 | DTBRVC |
| Bug 修复 | TDD 修复（先写重现测试） |

## 四、关键经验

### 4.1 UI 一致性是最容易腐化的

**问题**：功能越做越多，每个功能的 UI 样式略有不同，最终变成大杂烩。

**解决方案**：
1. 所有样式定义在共享文件中（如 BrandColors.xaml）
2. 禁止内联写死任何 UI 属性
3. Verify 阶段有 UI 一致性检查清单
4. 新 UI 模式必须提升为共享样式

**核心规则**：`design-guide.md` 是唯一的设计真相来源。

### 4.2 进度中断恢复是必须的

AI 对话有上下文限制，一个功能可能跨多次对话。没有进度文件，每次都要从头开始。

**实现方式**：`feat-progress.json` 记录当前阶段、设计方案、已创建文件。每个阶段完成后立即更新，完成功能后删除。

### 4.3 /prd 和 /feat 的职责边界

**最初的错误**：让 /prd 生成 CLAUDE.md。结果 /init 也会生成，两边冲突。

**最终方案**：
- `/prd` 只生成 PRD.md + 项目配置文件
- `CLAUDE.md` 由 /init 管理
- `testing-guide.md` 是通用文件，不由 /prd 生成

**经验教训**：
> 每个文件只有一个"owner"。如果两个流程都会修改同一个文件，设计就有问题。

### 4.4 AI 辅助开发的测试规则

AI 同时写代码和测试的最大风险：**测试只验证"当前行为"而不是"正确行为"**。

防御措施：
1. 人类定义测试意图（Design 阶段）
2. AI 写测试代码
3. 人类审查每个 Assert
4. 集成测试优先，少用 Mock

### 4.5 不要过度设计技能系统

**最初的冲动**：想要技能覆盖一切——CI/CD、部署、代码审查...

**现实**：MVP 阶段，两个技能（/prd + /feat）就够了。技能系统本身也要遵循 MVP 原则。

等到真正遇到问题再加新技能，而不是预测未来需求。

## 五、踩过的坑

### 5.1 文件联动不一致

**现象**：`testing-guide.md` 在一个文件里标为"通用"，在另一个文件里标为"项目专属"。

**原因**：多次迭代修改时，只改了一处忘了另一处。

**解决**：每次修改后，用检查脚本验证所有文件的交叉引用一致性。

### 5.2 测试方法论混入项目配置

**现象**：`testing-guide.md` 里写着 `dotnet test` 命令和 FlaUI 配置。

**原因**：最初没有区分"方法论"和"配置"。

**解决**：测试方法论（分层策略、DBTVC 测试职责）→ `testing-guide.md`。项目测试配置（框架、命令）→ `project-config.md` 的"测试约定"章节。

### 5.3 UI 测试持续失败

**现象**：FlaUI UI 自动化测试因 Windows 安全策略（应用程序控制策略）始终失败。

**经验**：UI 自动化测试的环境依赖很强。在 `known-issues.json` 记录此类环境问题，不让它阻塞开发流程。

## 六、跨项目复制清单

把这套技能搬到新项目，按此清单操作：

```
□ 1. 复制 .claude/skills/ 到新项目
□ 2. 运行 /prd，对话确定新项目的需求和技术栈
□ 3. /prd 自动生成:
     - docs/PRD.md
     - .claude/skills/feat/project-config.md
     - .claude/skills/feat/design-guide.md
□ 4. （可选）运行 /init 生成 CLAUDE.md
□ 5. 运行 /feat 开始开发
□ 6. 验证：
     - /feat 能读到 PRD.md？
     - /feat 能读到 project-config.md？
     - 构建命令是否正确？
     - 测试命令是否正确？
```

## 七、TDD 重构经验

### 7.1 从 DBTVC 到 DTBRVC 的核心变化

```
传统:  Design → Build → Test    → Verify → Commit
TDD:   Design → Test  → Build → Refactor → Verify → Commit
```

本质区别不是"多了个 Refactor"，而是**Test 的位置变了**。Test 从"验证工具"变成了"设计工具"——先定义"对是什么样"，再去实现。

### 7.2 务实 TDD vs 严格 TDD

最重要的经验：**不要教条化**。

- 严格 TDD 适合：业务逻辑复杂的功能、数据持久化、核心流程
- 灵活处理适合：UI 探索、快速原型、需求模糊的功能

MVP 阶段的开发者不是测试工程师，强制每个功能都严格 TDD 会拖慢进度。关键是在**最有价值的地方**用 TDD（业务逻辑），在不值得的地方灵活处理（纯 UI）。

### 7.3 需求不清晰时的 TDD 处理

需求模糊不应该阻塞开发。处理方式：
- 确定的部分正常写测试
- 不确定的部分标记"待确认"，跳过
- 完全不确定的整个功能用 Build 先行探索

TDD 本身就是澄清需求的工具——写不出测试意图的地方，就是需求没想清楚的地方。

### 7.4 AI 辅助 TDD 的三个关键原则

1. **测试意图 AI 设计，用户审查**：用户不需要懂测试工程，只需要确认"这些是我想要的功能行为吗？"
2. **上下文污染在 TDD 中天然被解决**：Test 先于 Build，写测试时实现代码不存在。但 Bug 修复时要注意——先描述正确行为，再写断言。
3. **两顶帽子**（Martin Fowler）：Green 阶段只管通过测试（允许丑代码），Refactor 阶段只管代码结构（不加功能）。这个分离对 AI 也有效——约束 AI 每个阶段只做一件事。

### 7.5 测试也可以删除

Kent Beck 的观点：测试不是越多越好。如果同一个行为被多种方式测试，那是耦合，耦合有维护成本。

删除场景：冗余测试、被更高层测试覆盖的旧测试、闪烁测试（不稳定的）。
底线：删除后核心业务逻辑的覆盖率不能降低。

### 7.6 大局观检查

Uncle Bob 提醒：严格的小步 Red-Green-Refactor 可能导致局部最优。每 3-5 个循环后暂停，检查整体架构是否合理，有没有"为了通过测试的 hack"需要替换。

## 八、给新手的建议

如果你想搭建自己的 AI 辅助开发技能系统：

1. **从最简单的版本开始**：一个 skill.md 文件就够了，不要一开始就追求完美架构
2. **在实际项目中迭代**：用起来才知道哪里不好用，不要空想
3. **通用 vs 专属要分清**：最常见的错误就是把项目配置写进通用流程
4. **每个文件一个 owner**：避免两个技能修改同一个文件
5. **技能系统本身也是 MVP**：不要过度设计，够用就行
6. **记录经验**：每次迭代的决策和原因都值得记录（比如这篇文档）
